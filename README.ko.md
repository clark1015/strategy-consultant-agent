# 🧭 Strategy Consultant Agent

[English](README.md) | **한국어**

**답을 주는 대신, 당신을 인터뷰하는 AI 전략 컨설턴트.**

대부분의 AI "비즈니스 어드바이저"는 뻔한 SWOT을 던져놓고 끝낸다. 이 에이전트는 정반대로 움직인다. **소크라테스식, 맥킨지 스타일 인터뷰**를 진행한다: 한 번에 한두 개의 날카로운 질문을 던지고, 당신의 답을 구조화해서 되돌려주고, *진짜* 문제가 정의되기 전까지는 다음 단계로 넘어가지 않는다. 인사이트는 *당신의* 입에서 나온다. 에이전트는 당신이 단계를 건너뛰지 않게 지킬 뿐이다.

**이 리포의 정체:** 컨설팅 방법론 — 구조적·가설 기반 문제 해결 — 을 *작동하는 [Claude Code](https://docs.claude.com/en/docs/claude-code) 에이전트로 엔지니어링한* 실전 예제. 복사해 붙이는 프롬프트가 아니라, 함께 생각하는 에이전트다: 페르소나(`CLAUDE.md`) + 워크플로우 스킬 2개 + 증류된 방법론 라이브러리. 흥미로운 건 프레임워크가 아니라, *사고방식*이 실행 가능한 무언가로 컴파일됐다는 점이다.

## 왜 만들었나

사이드 프로젝트를 실서비스로 만드는 과정에서 의사 결정의 어려움을 느끼던 답답함에서 출발했다.

코딩 에이전트가 세상에 나온 후 코딩은 더이상 병목이 아니었다. 어려운 것은 코드 *앞에 놓인* 결정들이었다: 어떤 사용자의 고통을 먼저 풀 것인가, 이 기능은 주류인지 곁가지인지, 내가 꽂힌 이 문제는 애초에 실재하는지. 돌아보면 나는 관성과 감으로 결정하고 있었다. 코드에는 타입, 테스트, 리뷰가 있어 내 실수를 잡아준다. 하지만 의사결정 과정에는 이를 디버깅하고 검토하는 장치가 존재하지 않았다. 

내가 원했던 건 맥킨지 같은 컨설팅펌이 제도화한 규율이었다. **구조적·가설 기반 문제 해결**. 어떤 옵션도 검토 없이 넘어가지 않게 하는 MECE 분해. 모든 주장을 증명하거나 죽일 수 있게 만드는 결론 형태의 가설. 문제가 정말로 정의되기 전엔 다음으로 넘어가지 않는 파트너.

그래서 방법론에 대한 공개 자료를 파고들었다. 내 언어로 다시 써서 `concepts/` 라이브러리로 증류했고, 진짜 결정이 닥칠 때마다 마주 앉을 에이전트로 만들었다. "사업계획서 써줘" 봇이 아니다. 컨설팅급 엄밀함으로 나를 대신 생각해주는 게 아니라 나 스스로 생각하게 만드는 인터뷰어다. 내가 실제로 행동에 옮기는 결론은 내 입에서 나온 것뿐이니까.

## 무엇이 다른가

| 흔한 "AI 어드바이저" | 이 에이전트 |
|---|---|
| 답을 준다 | *당신이* 답을 찾을 때까지 질문한다 |
| 프레임워크 템플릿을 채운다 | 프레임워크로 **인사이트**에 도달하면 멈춘다 |
| 곧장 해결책으로 점프한다 | *문제 정의*가 단단해지기 전엔 떠나지 않는다 |
| 일회성 덤프 | 상태 유지: 상담이 저장되고 이어진다 |

설계의 베팅: **전략에서는 좋은 질문이 자신만만한 답을 이긴다.** 결론을 건네주는 컨설턴트는 당신을 의존하게 만들고, 당신의 사고를 심문하는 컨설턴트는 당신을 날카롭게 만든다.

## 90초 데모

> 대표적인 `/consult` 세션 — 아이디어는 가상, 흐름은 실제. 동남아 음식 배달 아이디어를 탐색하는 장면.

```
당신:     베트남에서 음식 배달 앱을 만들고 싶어요.

에이전트:  시장 크기를 재기 전에 — 이게 정확히 누구의 고통을 없애주나요?
          식당, 라이더, 소비자 중에요. 잃으면 울 것 같은 쪽 하나만 골라보세요.

당신:     소비자요. 주문이 너무 번거로워요.

에이전트:  "번거롭다" — 그건 시간 문제인가요, 돈 문제인가요?

당신:     시간이요. 근처에 괜찮은 걸 찾는 데 한참 걸려요.

에이전트:  좋아요. 구조화해서 돌려드릴게요: 당신의 문제는 "배달"이 아니라
          발견(discovery) — 소비자가 좋은 음식을 빨리 못 찾는다는 것.
          동의하세요?

당신:     ...어, 그러네요.

에이전트:  어디서 싸울지 정하기 전에 — 소비자의 여정을 빈틈없이 쪼개봅시다:
          당긴다 → 검색 → 결정 → 주문 → 수령. 이 사이로 빠지는
          단계가 있나요?

당신:     아뇨, 다 덮어요. 그리고 검색 → 결정에서 부러져요.

에이전트:  좋습니다 — 어느 문제를 공략할지 찍은 게 아니라, 나머지를
          배제해낸 겁니다. 그게 MECE 규율이에요: 안 살펴본 경우를
          남기지 않기.

에이전트:  그럼 함정 하나: 배달은 이미 Grab과 ShopeeFood가 꽉 잡고 있어요.
          당신의 쐐기가 '발견'이라면, 그건 완전히 다른 싸움입니다.
          증명하거나 죽일 수 있는 주장으로 말해보세요 — "___는 참이다."

당신:     배달이 포화된 곳에서도 발견은 미충족이지 않을까요?

에이전트:  그건 질문이지 가설이 아니에요. 평결로 커밋하세요.

당신:     발견이 진짜 미충족 니즈다 — 이걸 푸는 게 4번째 배달 앱보다 낫다.

에이전트:  이제 검증할 게 생겼네요. 이게 참이라는 걸 알려면 시장에서
          무엇을 봐야 할까요? (그게 당신의 Key Question입니다.)
```

주목할 점: 에이전트는 한 번도 "여기 사업계획서요"라고 말하지 않았다. 막연한 아이디어 → 날카롭고 반증 가능한 가설 → 검증 질문으로, 한 번에 한 걸음씩 사용자를 옮겼을 뿐이다.

## 방법론: CSAC

모든 상담은 네 단계를 따른다. 에이전트는 항상 자신이 어느 단계에 있는지 알고 있고, 건너뛰게 두지 않는다.

```
Clarification  →  Structuring   →  Analysis    →  Conclusion
 (진짜 문제?)      (분해하고        (가설별 검증     (다음에 뭘
                   가설 세우기)      설계)           할 것인가)
```

- **Clarification(명확화)** — 범위, 용어, 전제가 고정될 때까지 문제를 재정의한다. "이게 애초에 진짜 문제인가?"
- **Structuring(구조화)** — 문제를 MECE하게 분해하고, 사실을 *결론 형태의 가설*("X는 참이다!" — "X는 참인가?"가 아니라)로 밀어올린다.
- **Analysis(분석)** — 가설마다 **Key Question**(무엇이 이를 증명하는가?)과 **Answer First**(답이 어떤 모양의 데이터에 들어 있는가?)를 설계한다. 설계가 서면 에이전트가 직접 검증 리서치를 돌린다 — 결과는 항상 *사실 → 가설에의 함의* 형식으로, 데이터 덤프 없이.
- **Conclusion(결론)** — 검증된 발견을 한 문단의 전략과 2~3개의 구체적 다음 행동으로 수렴시킨다 — 원하면 결론 우선 1장 메모(피라미드 원칙 + SCQA)로 정리한다.

방법론의 핵심 스탠스:

> *"최선의 전략은 가장 후회가 적을 전략이다 — 즉, 고르기 전에 모든 옵션을 살펴봤고, 다 살펴보기 전엔 넘어가지 않았다는 뜻이다."*

## 무엇을 함께 생각할 수 있나

"내 스타트업 아이디어 검증해줘"만이 아니다. 단계를 건너뛰는 것 자체가 리스크인 모든 결정의 구조화된 사고 파트너다:

- **새 제품·서비스 아이디어** — 만들기 전에 압박 테스트.
- **서비스·기능 방향 결정** — "A로 갈까 B로 갈까?"를 감이 아니라 MECE하게 추론.
- **시장 진입·포지셔닝** — 실제로 이길 수 있는 전장 찾기.
- **제안서·피치** — `/pitch-review`로 7개 렌즈에 대고 스트레스 테스트.

매번 일은 같다: 당신이 단계를 건너뛰지 않게 지키고, 결론이 *당신에게서* 나오게 하는 것 — 그래야 그 결론을 소유하고, 방어할 수 있으니까.

## 안에 뭐가 있나

```
CLAUDE.md            컨설턴트 페르소나. Claude Code에 넣는 순간
                     답변 기계가 아닌 인터뷰어가 된다.
concepts/            방법론을 7개의 레퍼런스 노트로 증류
                     (MECE, CSAC, 초기 리서치, Key Question, 프레임워크,
                     피라미드 메모…).
.claude/skills/
  consult/           /consult — 새 아이디어에 풀 CSAC 인터뷰 실행.
  pitch-review/      /pitch-review — 피치 덱을 7개 렌즈로 스트레스 테스트.
cases/               저장된 상담 기록 (살아있는 문서, 이어서 진행 가능).
```

## 써보기

1. [Claude Code](https://docs.claude.com/en/docs/claude-code)를 설치한다.
2. 이 리포를 클론하고 연다: `cd strategy-consultant-agent && claude`.
3. 아이디어를 그냥 말하거나 — `/consult`로 구조화된 인터뷰를 시작한다.

리포 루트의 `CLAUDE.md`가 Claude를 자동으로 컨설턴트 모드로 전환시킨다.

## 출처와 범위

방법론은 전략 컨설팅의 기본기(구조적 문제 해결, 가설 기반 사고, MECE) — 탑 컨설팅펌들이 가르치는 바로 그 툴킷 — 를 **내 언어로 재구성해** 증류한 것이다. 이 리포에 담긴 것은 *나의* 재서술과 에이전트 설계이지, 제3자의 슬라이드나 저작물이 아니다.

어떤 컨설팅펌과도 제휴·보증 관계가 없다.
