# CSAC — the problem-solving arc

Four stages, run in order. The agent always knows which one it's in, and won't let you skip ahead.

| Stage | Does | Produces |
|---|---|---|
| **C**larification | Re-define the problem, set the terms | A sharp problem statement |
| **S**tructuring | Break down MECE-ly, find priorities, form hypotheses | A prioritized hypothesis list |
| **A**nalysis | Design Key Questions, verify them | Validated findings |
| **C**onclusion | Integrate findings into strategy | A recommendation + next actions |

The working order in practice: clarify → tidy up the clarification → note what needs research + do initial research → summarize the situation → form hypotheses → derive Key Questions → deep research to verify → draw insight → write the flow. **Always know where on this line you are.**

## Clarification — re-ask the real problem

The first act of thinking is asking **"is this actually the problem?"** This stage decides *what* to analyze — it isn't the analysis yet.

Five things to pin down:

1. Background and context.
2. **Definitions of terms** — e.g. "sales are slowing": in absolute value? growth rate? relative to competitors?
3. Scope and its limits (time, geography, segment).
4. Authority and available resources.
5. Validating questions.

Ask in four passes: **big picture → specific context → detail → validation.** It doesn't matter which pass a given question belongs to; what matters is *drilling down until the vague becomes fixed*, producing a clear frame and a clear scope.

**Can't ask the questions?** (No access to the client, no time.) Then: (a) ask directly if you can, (b) infer from public info / initial research, or (c) **organize it internally and adopt it as an assumption.** Clarification never fails just because no one answers you — its essence isn't the questions, it's *building your own frame for looking at the problem.* Mark assumptions so you can overturn them later.

## Structuring — decompose and hypothesize

Grasp the structure broadly, then pick the priority tasks to solve.

Key moves: decompose the problem → keep it MECE → reshape it (via logic tree or framework) into **sharp hypotheses you can test right away** → prioritize.

A framework is only a means to MECE structure — any lens fits if it fits the situation, including one you invent. See [[05-frameworks]].

In theory you list and test *every* hypothesis. In practice you **prune to the clear priorities** using what Clarification already settled, the client's constraints, and signals from initial research (how often something shows up in coverage, in reports).

## Analysis — verify through Key Questions

Decompose down the chain: **hypothesis → (so what?) → Key Question → (so what?) → Sub Question** (a concretely researchable piece).

Never lose the thread: *"which hypothesis am I testing, to reach which conclusion?"* See [[04-key-questions]].

## Conclusion — the final output

Produce the recommendation. Combine the validated Key Questions into a storyline and an insight. For how a strong proposal is built, see [[06-proposal-anatomy]]; to write the result down as a one-page memo, see [[07-pyramid-principle]].

## A worked example (compressed)

*An investor is weighing a mobile-phone retail play in an unfamiliar market.*

Clarify by splitting the questions into three MECE areas (client's definition / market / the investment) → get answers → use a **Value Chain** to decompose the retail structure → spot that "the only lever we can actually adjust is the point of sale" → split that into internal/external → a client constraint pushes focus internal → decompose the buyer's journey into stages → two insights (the salesperson's influence is large; promotions are timely) → five concrete investment actions at the recommend / plan / after-sales stages.

The shape to notice: **decompose → prune → focus → insight → action.**
