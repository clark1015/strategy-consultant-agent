# Key Questions & Answer First

How to verify a hypothesis fast, without drowning in research.

## Key Question (KQ)

Not just "something you're curious about" — **the question whose answer proves or kills a hypothesis.**

Think of a KQ as **the best conclusion you can pre-write given what you know now.** Stack enough validated KQs and you get a storyline, then an insight.

The KQ ↔ Answer loop: going KQ → Answer is **So what?** (the clear answer to the hypothesis); going Answer → KQ is **Why so?** (why does this matter?).

### Two tests for a good KQ

1. **Does it descend from the hypothesis?** Is it actually needed to prove/disprove the claim?
2. **Is it useful after you answer it?** Once answered, does it let you *decide* something downstream?

### Four ways a KQ goes wrong

Hypothesis: *"Revenue fell because active users stopped growing."*

| Bad KQ | Why it's bad |
|---|---|
| What's the new-signups trend? | Meaningful, but off the verification target |
| Why did last month's revenue drop? | Already known, or something another KQ will cover |
| What's the competitor's user trend? | No direct link to the hypothesis |
| Should we skim the user reviews? | No clear purpose, no direction |

### KQ → SQ

Break a KQ into concretely researchable Sub Questions.

*Hypothesis:* a branch-centric sales channel is optimized for older customers, so it fails to capture the 20–30s. → *KQ:* is the current channel structure designed around older customers? → *SQ:* what % of contracts come through branch visits? is the digital-channel share low?

## Answer First (AF)

Not "decide the answer you want." It's **fixing in advance the *form* of the answer and the data it lives in — and imagining the insight it would yield — before you go looking.**

The hypothesis–KQ–AF chain:

> **[Hypothesis] → [what I'd learn if it's true = KQ] → [the evidence I go find = AF]**

- Forward: "I'm looking for *this* evidence in order to test *this* hypothesis" — gives the search purpose.
- Backward defense: if I *can't* find evidence of this shape, the hypothesis is moot. This **stops you wandering because 'the data isn't there,'** saving time and mental energy.

Proving a hypothesis "true" with just any evidence is worthless — if it's true, it has to be **usable in the strategy.**

### A worked example (compressed)

*Problem: a brand isn't winning in the golf-apparel market.*

- Direction 1 — "poor fit with demand/positioning"
  - H1 positioning mismatch → *KQ:* what perception do buyers hold? → *AF:* brand-funnel study, positioning map by age/gender
  - H2 wrong segment → *KQ:* what drives each segment's purchase? → *AF:* consideration/conversion by segment, social-topic analysis
  - H3 local expectations unmet → *KQ:* what are the regional apparel trends? → *AF:* critic interviews, best-seller comparison by region
- Direction 2 — "poor product/lineup fit"
  - H4 collection identity → *KQ:* are the price tiers and line structure clear? → *AF:* line/price-tier map, styling-suggestion depth

*Result:* H1 true / H2 **no info** (a hypothesis you can't verify is moot) / H3 true / H4 false → findings 1+3 converge upward → "we need a strategy that fixes the positioning mismatch and the local-expectation gap."

## Planning and running it

- **Plan:** to fill the blanks, prioritize the sources and frames you'll search. Once you've fixed *why* you're searching and *what proving it changes*, research collapses to **just checking whether the info exists** — purpose-driven, efficient.
- **Every verdict is one of three: true / no info / false.**
- **KQ filter (3 tests):** does verifying it (1) contribute to solving the problem, (2) touch the core of *this* problem, (3) carry an interesting insight? Pass the KQs through, then draw insight — individually or by connecting them. When findings converge upward, a new vantage point appears one level up, yielding a broader insight.
