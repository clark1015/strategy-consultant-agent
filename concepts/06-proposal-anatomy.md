# Anatomy of a Strong Strategy Proposal

What separates a proposal that persuades from a deck that merely informs. These seven lenses double as the review checklist for `/pitch-review`.

## The logical spine

A strong proposal tends to run this shape:

1. **Premise** — declare the core value up front, plus the growth logic (what the *leading variable* is).
2. **Problem definition** — show where the need currently lives, and why the incumbents are *unfit* for it, then **redraw the battlefield** onto the axis you can win.
3. **Direction** — map the customer/usage cycle to your capabilities; branch a logic tree; name the targets; state the concept.
4. **The strategies** — a small set, each on an identical template (overview → concrete steps → expected effect).
5. **Integration** — a roadmap that ties the strategies to a real timeline, closing on the same line the concept opened with.
6. **Appendix** — the proof of how the core value was derived, kept out of the main story.

## Seven review lenses

1. **Conclusion first, proof after.** Declare the core value as a premise; move the derivation to an appendix. The main body is a *story*; the appendix is the *proof*. A reader should get the point before they get the evidence.

2. **Problem definition = choosing the battlefield.** Don't fight the incumbent where it's strong. Redraw the problem onto an axis where you win, and let your comparison table show *only the axes you can win on*.

3. **A logic-tree split that becomes the sorting axis for the strategies.** The branches (e.g. *expand awareness* vs. *drive directly*) should be the same axis that organizes the strategy list — and the set should close the funnel (acquisition + retention), not just open it.

4. **1:1 alignment of target × strategy.** The targets you name early should be the *same* ones reused in the scenarios and strategies later. Reuse is the consistency device that makes a document feel like one argument, not many.

5. **A standard slide grammar.** Top: a one-sentence headline (the conclusion). Middle: the evidence (comparison table, diagram, real screenshot). Bottom: a one-sentence implication. Repeat one template across the strategy section so a reviewer can compare every strategy on the *same* yardstick.

6. **A concept that bookends.** Open and close on the same line, varied by target in between. Bookending makes the whole thing feel authored, not assembled.

7. **Signals of feasibility.** Design the ask *as* product usage (signing up = using the thing), cite real benchmarks, show your own real usage, and base the roadmap on the target's actual lifecycle. These are what defend against the "nice idea, won't ship" critique.

## One caution

A proposal can be almost entirely qualitative — built on structural analysis (feature comparison, UI teardown, positioning maps) rather than market-size and share numbers. That's **framework-driven + lived-experience-driven**, not data-driven, and it can be completely valid. But when the ask *requires* quantitative grounding, you have to lay deep-research data on top of this structure — the structure alone won't carry it.
